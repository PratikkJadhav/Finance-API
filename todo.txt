Here's the exact order:

**1. Project scaffold + go.mod**
Just the folder structure and `go mod init`. Nothing runs yet, but you have a skeleton to fill in. 10 minutes.

**2. Database schema + migrations**
Write your SQL — `users`, `transactions` tables with all columns, indexes, constraints. Use `golang-migrate` to manage it. This is the foundation everything else touches.

**3. DB connection + repository layer**
Connect to Postgres with `pgx`, write your first raw query (e.g. `CreateUser`). Make sure data flows in and out correctly before touching HTTP at all.

**4. Auth — register + login**
Hash password with bcrypt on register, generate JWT on login. No middleware yet, just the logic working end to end.

**5. Auth middleware + role middleware**
`AuthMiddleware` that parses the JWT and injects user into context. Then `RequireRole(...)` that reads from context and returns 403 if wrong role.

**6. Transaction CRUD handlers**
Now wire up the HTTP layer — handlers call services, services call repositories. With middleware already done, you just attach it to routes.

**7. Dashboard/summary endpoints**
The aggregation queries — total income/expense, monthly trends, category breakdown. Pure SQL, no Go logic needed.

**8. Validation + error handling**
Go back through all handlers and make sure bad input returns proper 400s with useful messages.

**9. README + Design Decisions section**
Write it last when you actually know what tradeoffs you made.

Here are the most relevant repos, ranked by how closely they match your exact stack:

---

**Closest match to your exact stack (Go + Chi + pgx + JWT + RBAC)**

**[github.com/dhax/go-base](https://github.com/dhax/go-base)**
Go RESTful API boilerplate with JWT Authentication backed by PostgreSQL. Has `/auth` routes plus separate `/api` and `/admin` routes, with the admin route requiring an administrator JWT in the Authorization header. This is the closest to what you're building — Chi, Postgres, JWT, role-differentiated routes.

---

**Clean architecture reference (Chi + JWT + Docker)**

**[github.com/altafino/go-backend-clean-architecture-chi](https://github.com/altafino/go-backend-clean-architecture-chi)**
Chi, JWT auth middleware, bcrypt, testify, Docker. Clean folder separation: `api/controller`, `api/middleware`, `api/route`, `domain`, `repository`. Good for seeing how to wire the layers together — though it uses MongoDB, so ignore the DB layer and borrow the structure.

---

**Raw SQL + pgx without ORM (exact philosophy match)**

**[github.com/pauljamescleary/go-rest-postgres](https://github.com/pauljamescleary/go-rest-postgres)**
No ORM — uses pgx directly with scany for struct mapping. Explicitly argues that declarative SQL in code is preferable to code generation and ORM APIs. The README even explains *why* each choice was made — good README inspiration for your submission.

---

**Fully fleshed Go + Chi + Postgres starter kit**

**[github.com/gmhafiz/go8](https://github.com/gmhafiz/go8)**
Go + Postgres + Chi Router + sqlx + authentication + testing + OpenTelemetry. Heavier than what you need, but useful to see how auth middleware, dependency injection, and layering work together at scale.

---

**My recommendation:** use `dhax/go-base` as your structural reference for routing + JWT + role separation, and `pauljamescleary/go-rest-postgres` for the raw SQL + pgx patterns. Between those two you have everything you need without copying any one repo wholesale — which is exactly the right way to use references for an eval submission.
